# 多文件合并后台解析设计

## 背景

多文件合并在粘贴多个 Excel 后，会在页面主线程中同步调用 SheetJS 的 `XLSX.read()` 和 `sheet_to_json()`。复杂工作簿连续解析时会阻塞事件循环，导致上传进度停止、超时计时器失效，并触发浏览器“此页面没有响应”提示。

## 目标

- 文件选择、拖拽和粘贴统一使用后台解析路径。
- 连续处理多个复杂 Excel 时，页面保持可交互并持续显示逐文件进度。
- 单个文件读取、解析失败或超时只影响该文件，后续文件继续处理。
- 保持浏览器本地离线处理，不增加服务器、外部网络依赖或 API Key。
- 浏览器未提供有效剪贴板文件时给出明确提示，不再静默无响应。

## 非目标

- 不重构为框架项目。
- 不改变合并、Sheet 选择、预览和导出结果的数据结构与行为。
- 不承诺浏览器无法暴露系统剪贴板文件时仍能强制读取；此时引导用户拖拽或选择文件。

## 方案

### Worker 架构

给现有内嵌 SheetJS `<script>` 增加稳定标识。首次解析时读取该脚本内容，与一段最小 Worker 消息处理代码组合为 Blob URL，并创建专用 Web Worker。

主线程负责：

1. 从 `File` 异步取得 `ArrayBuffer`。
2. 通过 transferable 把缓冲区交给 Worker。
3. 接收工作簿名称、Sheet 名称和二维数据。
4. 更新现有 `multiFiles` 状态和页面进度。

Worker 负责：

1. 运行已内嵌的 SheetJS，不请求网络资源。
2. 调用 `XLSX.read()`。
3. 对每个 Sheet 调用 `sheet_to_json()`。
4. 返回可结构化克隆的数据或精简错误信息。

### 队列与超时

- 所有文件顺序解析，避免多个大型工作簿同时占用大量内存。
- 文件读取维持独立超时。
- Worker 解析超时设为 60 秒。超时后终止 Worker、释放 Blob URL，并在下一文件前重新创建 Worker。
- Worker 报错、消息错误或超时都统一转换为当前文件的读取失败，不中断整个队列。
- 每个文件完成后立即渲染列表和进度，让用户能看到持续推进。

### 粘贴错误反馈

多文件合并页收到粘贴事件时：

- 有受支持文件：阻止默认粘贴，进入统一解析队列。
- 有文件条目但格式不受支持：提示仅支持 `.xlsx`、`.xls`、`.csv`。
- 没有文件条目：提示浏览器未提供剪贴板文件，请改用拖拽或“选择文件”。

## 错误与资源清理

- 用户离开页面或清空文件时，不强制中断已开始的单个解析；解析完成后按现有状态逻辑处理。
- Worker 超时或异常必须 `terminate()`，并撤销对应 Blob URL。
- 错误提示包含文件名和类别：读取超时、解析超时、文件格式异常或浏览器不支持 Worker。
- 不在控制台输出文件内容或单元格数据。

## 测试

先添加失败的回归测试，再实现：

1. 多文件解析不再直接在主线程调用 `XLSX.read()`。
2. Worker 使用页面内嵌 SheetJS，且不包含外部 URL。
3. Worker 超时会终止实例并拒绝当前文件。
4. Worker 失败后队列继续处理下一文件。
5. 无有效剪贴板文件时出现明确提示。
6. 现有按同名 Sheet 分组和导出回归测试继续通过。
7. `index.html` 全部脚本通过 `new Function` 语法检查，`excel-tools.html` 仍只负责跳转。
8. 本地静态服务中验证：标准剪贴板 CSV、四个较大文件、异常文件混合队列，以及控制台无相关错误。

## 验收标准

- 粘贴四个约 2.7MB 的复杂 Excel 后，页面不会出现“没有响应”。
- 页面逐文件显示进度，并能在解析期间响应主题按钮或其他轻量交互。
- 单个文件在 60 秒内无法完成时被跳过，其余文件继续。
- 不支持的剪贴板内容获得可操作的提示。
- 所有处理仍在浏览器本地完成。
