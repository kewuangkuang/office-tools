# 回单模式结构化改造设计

## 目标

把 PDF 回单模式从两个大型 HTML 文件中可漂移的复制逻辑，收敛为一套可定位、可测试、可回退的流程；优先保证扫描件的裁剪、金额、付款人、收款人识别。

## 范围

只处理回单模式：上传 PDF、页面渲染、回单裁剪、OCR、字段解析、文件名与结果展示。多文件合并、Sheet、重命名等其他工具不改。

## 当前已证实的问题

`index.html` 与 `excel-tools.html` 分别复制了一套完整应用。两者函数数量不一致，且曾出现 OCR 输入不同：本地入口先对裁切图做去边和二值化，在线入口直接识别原图。扫描回单的付款/收款小字因此在本地入口丢失。

## 设计

### 1. 一个应用来源，保留兼容入口

`index.html` 是唯一的应用文件和唯一允许承载业务代码的入口。`excel-tools.html` 保留文件名，仅跳转到同目录的 `index.html`；它不再包含第三方库、页面样式或业务函数。

这既兼容已有本地文件、书签和旧链接，也彻底消除双入口漂移。以后所有功能修改只改 `index.html`。

### 2. 建立回单功能地图与入口契约

新增回单模式说明，固定六个边界：

1. PDF 页面渲染
2. 扫描件/文字层裁剪
3. OCR 调度
4. 字段解析
5. 结果命名
6. 预览与导出

每个边界记录入口函数、输入、输出和验证用例。测试同时断言两个 HTML 的关键调用契约一致，避免再次只修复一个入口。

### 3. 逐步收敛共用逻辑

不一次性重写整个网站。先抽出仅供回单模式使用的纯函数：裁剪策略、OCR 文本修正、字段解析、字段合并。两个入口以相同脚本加载这组纯函数；页面 DOM 和各工具状态仍留在原 HTML。

这会消除 OCR 与解析逻辑的双份复制，同时将风险限制在回单模式。

### 4. 真实文件验证

“丽水.pdf”作为扫描件回归样本：前两页双回单、第三页单回单；预期 5 个结果、没有残片、每张结果有金额和两方字段或明确 OCR 警告。

“五一.pdf”作为文字层样本：优先使用文字层，验证金额和两方字段，不触发扫描件 OCR。

## 不做的事

- 不上传 PDF，不接入百度 OCR 或任何后端。
- 不重写其他工具页，不迁移框架。
- 不为某一家公司的名称写死补丁；只修正通用 OCR 标签误读和布局规则。

## 验收标准

- 两个入口对回单 OCR 都使用原始裁切图。
- `excel-tools.html` 不含完整应用或回单业务函数，并会跳转到 `index.html`。
- 丽水扫描件裁切为 5 张，不含第六张残片。
- 真实 Tesseract 输出经字段解析后付款人和收款人不为空。
- 所有回单回归测试、两份 HTML 的内嵌脚本语法检查通过。
- 每项改动有本地 Git 提交，可回退。
