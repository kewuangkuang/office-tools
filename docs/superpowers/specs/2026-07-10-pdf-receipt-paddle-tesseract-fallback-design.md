# PDF 回单识别双引擎回退设计

## 基线

- PaddleOCR 基线：GitHub 提交 `9a061809002d`。
- Tesseract 基线：GitHub 提交 `cae519c03f30`。
- 不采用后续提交 `940d7bc3a1e8` 和 `17f8f453efb2` 中重新拼装的 OCR 主流程。

## 目标

回单模式优先使用 PaddleOCR 识别扫描件或图片型 PDF。当 PaddleOCR 未加载、初始化失败、识别报错、没有返回有效文字，或未识别出当前命名方式所需的关键字段时，自动使用 Tesseract 补充识别。

## 数据流

1. 使用 PDF.js 在浏览器本地读取 PDF 文字层并生成回单裁剪图。
2. 先从文字层提取金额、付款人、收款人、日期和流水号；已有结果保留。
3. 缺少当前命名方式需要的字段时，对该张回单裁剪图运行 PaddleOCR。
4. 合并 PaddleOCR 结果，只填充仍为空的字段。
5. 若所需字段仍缺失，或 PaddleOCR 不可用，则运行 Tesseract。
6. Tesseract 沿用 `cae519c03f30` 的渲染分辨率、裁剪、预处理、语言和字段提取策略，只补充空字段，不覆盖前面已经识别出的字段。
7. 两个 OCR 引擎都无法补齐字段时，保留可编辑结果并提示用户人工核对。

## 回退判定

回退以当前命名方式为准：

- 金额：金额为空时需要 OCR 或回退。
- 付款人户名：付款人为空时需要 OCR 或回退。
- 收款人户名：收款人为空时需要 OCR 或回退。
- 收款人户名加金额：任一字段为空时需要 OCR 或回退。
- 自定义：只检查模板实际引用的字段；`{序号}` 不需要 OCR。

此外，OCR 返回为空、只有极少无意义字符、初始化失败或运行异常，均视为该引擎识别失败。

## 资源与隐私

- 所有 PDF 渲染和 OCR 均在浏览器本地执行，不上传文件。
- 保留两个入口：`index.html` 与 `excel-tools.html` 必须同步。
- 在线 OCR 资源加载失败不能阻断拆分；应继续尝试下一引擎，并给出明确状态提示。

## 验证

- 为引擎选择、字段完整性判断和“只补空字段”写可重复执行的脚本测试，并先验证测试在实现前失败。
- 用文字层 PDF 验证无需 OCR 时不会启动 OCR。
- 用扫描版/图片版 PDF 验证 PaddleOCR 优先，Paddle 失败时 Tesseract 自动接管。
- 验证金额、付款人、收款人及自定义命名字段。
- 对两个 HTML 执行内嵌脚本语法检查，并确认相关 OCR 函数与导出路径保持同步。
